# OpenClaw VM プロジェクト — 背景・目的

## プロダクト概要

**プロジェクト名**: OpenClaw VM 環境構築

**一言で言うと**: OpenClaw（オープンソースの自律型AIエージェント）をVM上に安全に構築し、メッセージングアプリ経由でのAIアシスタント運用・スキル開発・アーキテクチャ学習を行うための環境設計

---

## 背景・課題

### OpenClaw とは

OpenClaw（旧称: Clawdbot / Moltbot / Molty）は、オーストリアのソフトウェアエンジニア **Peter Steinberger** が開発した
**オープンソースの自律型AIエージェント**。

- **公式サイト**: https://openclaw.ai/
- **GitHub**: https://github.com/openclaw/openclaw （247,000+ stars, 47,700+ forks — 2026年3月時点）
- **npm**: `openclaw` パッケージとしてグローバルインストール可能
- **ランタイム**: Node.js（v22以上必須）
- **ライセンス**: オープンソース
- **作者**: Peter Steinberger（2026年2月にOpenAIへ移籍、プロジェクトはオープンソース財団へ移管予定）

### OpenClaw の特徴

- **自律型AIエージェント**: プロンプトを待つのではなく、自ら判断してタスクを実行する
- **メッセージングアプリがUI**: WhatsApp, Telegram, Slack, Discord, Signal, iMessage など10以上のプラットフォームに対応
- **ローカル実行**: ユーザーのマシン上で動作し、データはローカルに保存
- **マルチLLM対応**: Anthropic Claude, OpenAI GPT, DeepSeek, Gemini, Ollama（ローカルモデル）等
- **スキルシステム**: SKILL.md ファイルベースのプラグイン機構で機能拡張可能
- **CLIファースト**: MCP（Model Context Protocol）ではなく CLI を中心としたツール連携設計

### できること

| カテゴリ | 具体例 |
|---|---|
| メール管理 | 受信メール要約、返信ドラフト作成、重要メール通知 |
| カレンダー管理 | 予定追加、空き時間検索、ミーティング調整 |
| ブラウザ操作 | Web検索、価格比較、データスクレイピング |
| ファイル操作 | ファイル整理、ドキュメント要約、レシート→スプレッドシート変換 |
| 開発ワークフロー | GitHub PR通知、コードレビュー補助、cron定期実行 |
| モーニングブリーフィング | 天気・カレンダー・ニュースの朝の定時通知 |
| IoT連携 | Home Assistant 経由で照明・サーモスタット制御 |

Nvidia CEO Jensen Huang は OpenClaw を「これまでにリリースされた最も重要なソフトウェア」と評した。
2026年3月時点で150万以上のAIエージェントがユーザーによって構築されている。

### なぜVMに入れるのか

OpenClaw は強力な権限（メール、カレンダー、ブラウザ、ファイルシステム、シェル）を必要とするため、
**セキュリティ上、メインマシンでの直接実行はリスクが高い**。

- **Cisco のセキュリティ研究チーム**: サードパーティ製スキルによるデータ流出・プロンプトインジェクションを確認
- **AI研究者 Gary Marcus**: 「起こるべくして起こる災害」と警告
- **中国工業情報化部**: 一部のOpenClawインストールに「極めて高い」セキュリティリスクがあると警告

VM上に構築することで:

- **ホストOSとの隔離**: メール・ブラウザ等の権限がVM内に閉じる
- **スナップショットで実験**: スキル開発やサードパーティスキルの検証を安全に行える
- **被害の局所化**: 万が一のプロンプトインジェクション攻撃でもホストOSは安全
- **環境の再現性**: 構築手順をドキュメント化し、何度でも同じ環境を再構築できる
- **キルスイッチ**: 問題発生時にVMごと停止できる

## AS-IS（現状）

### 現在の状態

```
- OpenClaw に興味はあるが、安全に動かせる環境がない
- メインマシンで動かすのはセキュリティリスクが心配
- 5つのサブシステム（Gateway, Brain, Memory, Skills, Heartbeat）の全体像を把握していない
- スキルの開発・テストを気軽にできる環境がない
```

### 現状の問題点

| 問題 | 影響 |
|---|---|
| メインマシンにシェル・メール・ブラウザ権限を付与するのが怖い | 試せない |
| サードパーティスキルの安全性が不明 | プロンプトインジェクションのリスク |
| Node.js 22 以上の環境を既存マシンに入れたくない場合がある | 環境汚染 |
| 障害時にホストOSに影響が及ぶ可能性 | リスクが大きすぎる |

## セキュリティに関する注意

### 必須のセキュリティ対策

1. **VMで実行する**: メインマシンでは絶対に実行しない
2. **Gateway トークンを強力にする**: 長くランダムな文字列を設定
3. **Canvas Host を `127.0.0.1` にバインド**: デフォルトの `0.0.0.0` はインターネットに公開される
4. **最小権限の原則**: 必要なスキルの権限のみ有効にする
5. **サードパーティスキルは検証してから使う**: プロンプトインジェクションのリスクあり
6. **SSH トンネル経由でアクセス**: Web UI を直接公開しない
