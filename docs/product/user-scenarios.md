## シナリオ 1: まず動かす — Gateway 起動 & 初回対話

**ゴール**: VM上で OpenClaw Gateway を起動し、メッセージングアプリからAIエージェントと対話する

### やること

1. VM を構築し、Node.js 22+ をインストール
2. `npm install -g openclaw@latest` でインストール
3. `openclaw onboard --install-daemon` でオンボーディングウィザードを実行
4. LLM プロバイダー（Anthropic 等）の API キーを設定
5. メッセージングプラットフォーム（Telegram or Slack）を接続
6. メッセージを送る → エージェントが応答! 🎉

### 学べること

- OpenClaw の基本的なセットアップフロー
- Gateway の仕組み（ポート 18789 でリッスン）
- `~/.openclaw/openclaw.json` の設定構造
- `openclaw doctor` によるヘルスチェック

### 確認ポイント

- [ ] `openclaw --version` でバージョンが表示される
- [ ] `openclaw doctor` が全項目パスする
- [ ] Gateway がバックグラウンドで稼働している
- [ ] メッセージングアプリからエージェントに話しかけて応答がある

## シナリオ 2: メール管理 — 受信トレイを AI に任せる

**ゴール**: OpenClaw にメールの要約・分類・下書き作成を任せる

### やること

- Gmail / メールアカウントを OpenClaw に接続
- 「今日の未読メールを要約して」と指示
- 重要度に応じた分類ルールを設定
- 返信ドラフトの自動生成を試す

### 使い方の例

```
あなた: 「今週のメールで返信が必要なものをリストアップして」
OpenClaw: 3件の返信待ちメールがあります:
  1. [山田さん] ミーティング日程の確認 — 提案: 来週火曜14時
  2. [田中さん] 見積もり依頼 — ドラフトを作成しました
  3. [佐藤さん] プロジェクト進捗報告 — 要約を添付
  送信してよろしいですか？
```

### 楽しさ

- メッセージアプリから一言送るだけで受信トレイが整理される
- 返信ドラフトをレビュー→承認するだけで対応完了
- 本当に「AIが仕事をしてくれる」実感が得られる最初のシナリオ

## シナリオ 3: モーニングブリーフィング — 毎朝の自動通知

**ゴール**: 毎朝、天気・カレンダー・ニュース・メール概要を自動で受け取る

### やること

- カレンダー連携を設定
- Heartbeat（スケジューラー）で毎朝7時に定期実行を設定
- 通知内容をカスタマイズ（天気、今日の予定、未読メール件数、ニューストピック）

### 出力イメージ

```

🌡️ 天気: 東京 晴れ 最高18°C / 最低8°C

  - 14:00 クライアントミーティング（会議室A）

📧 未読メール: 12件（重要: 3件）
📰 注目ニュース: OpenClaw v2026.3.7 リリース — ContextEngine が大幅強化
```

### 学べること

- Heartbeat サブシステムの仕組み

## シナリオ 4: スキル開発入門 — SKILL.md を書く

**ゴール**: 自分だけのカスタムスキルを作成し、エージェントに新しい能力を追加する

### スキルの仕組み

OpenClaw のスキルは、フォルダ内の `SKILL.md` ファイルで定義される。
YAML フロントマター + 自然言語の説明 + ツール設定で構成される、非常にシンプルな仕組み。

### 作ってみるスキル例

**例: 読書メモ管理スキル**

```markdown
---
name: book-notes
description: 読書メモを管理する
triggers:
  - 読書メモ
  - 本のメモ
  - 読んだ本
---

# 読書メモ管理スキル

## やること
ユーザーが本について話したとき、以下を行う:
1. 書名・著者・感想をMarkdownファイルに保存
2. 過去のメモを検索・一覧表示
3. 月次の読書レポートを生成

## 保存先
~/openclaw-data/book-notes/ にMarkdownファイルとして保存

## フォーマット
- ファイル名: YYYY-MM-DD-書名.md
- 内容: 書名、著者、評価（★1〜5）、感想、印象に残った引用
```

### 学べること

- SKILL.md のフォーマットと構造
- スキルのトリガー設定
- ClawHub（スキルレジストリ）の仕組み
- エージェントへの自然言語による指示の書き方

### 確認ポイント

- [ ] 自作スキルフォルダが `~/.openclaw/skills/` に配置されている
- [ ] エージェントがスキルのトリガーワードに反応する
- [ ] スキルの指示通りにタスクが実行される

## シナリオ 5: ブラウザ自動化 — Web 操作をエージェントに委任

**ゴール**: Puppeteer/Playwright 経由でブラウザ操作を自動化する

- 「Amazon で4Kモニターの最安値を5つ探して」

## シナリオ 7: アーキテクチャ解析 — 5つのサブシステムを理解する

**ゴール**: OpenClaw の内部アーキテクチャを実際に動かして理解する

### 5つのサブシステム

```
OpenClaw Gateway (単一の Node.js プロセス)
├── Gateway    ... メッセージチャネル（Slack/WhatsApp等）からのメッセージルーティング
├── Brain      ... ReAct ループによる LLM 呼び出しオーケストレーション
├── Memory     ... Markdown ファイルベースの永続的コンテキスト管理
├── Skills     ... SKILL.md ベースのプラグイン機能
└── Heartbeat  ... タスクスケジューリング・受信トレイ監視
```

### 注目ポイント

| サブシステム | 学べること |
|---|---|
| Gateway | チャネル抽象化、メッセージングプラットフォームの統一インターフェース |
| Brain | ReAct パターン、LLM 呼び出しチェーン、モデルフォールバック |
| Memory | ローカルファイルベースのコンテキスト管理、セッション永続化 |
| Skills | プラグインアーキテクチャ、自然言語による機能定義 |
| Heartbeat | cron ベースのスケジューリング、イベント駆動設計 |

### やること

- `openclaw gateway --verbose` でデバッグモード起動し、ログを観察
- メッセージ送信時のログから Gateway → Brain → Skills の流れを追う
- `~/.openclaw/` 内の Memory ファイル（Markdown）を直接確認
- Heartbeat の設定を変更して定期実行の挙動を確認

### 学べること

- モノリシックだが疎結合なアーキテクチャ設計
- ReAct（Reasoning + Acting）パターンの実装
- ローカルファーストなAIエージェントの設計思想
- 「CLIがMCPに勝る」という設計哲学の実例

## シナリオ 8: 書類→Notion 自動振り分け（最優先ユースケース）

> **ステータス**: 検討中（2026-03-10 具体化開始）
> 詳細は `docs/ideas.md` の「優先ユースケース」セクションも参照

**ゴール**: 紙の書類をスマホで撮影し、Telegram 経由で OpenClaw に送ると、
OCR＋AI分類を経て Notion の適切なページに自動追加される。
結果は Telegram 上でレビュー・修正できる。

### ペルソナ

**自分（書類を受け取る親・生活者）**:
- 幼稚園のお手紙、病院の書類、保険・ローン関連の通知など紙の書類が日々届く
- Notion でカテゴリ別に管理したいが、手動の振り分けが面倒で溜まりがち
- スマホは常に手元にある。撮って送るだけで整理されてほしい

### シナリオ 8-1: 日常の書類スキャン（メインフロー）

**状況**: 幼稚園から行事予定表のお手紙をもらって帰ってきた

**シナリオ**:

```
1. スマホで書類を撮影し、Telegram の OpenClaw チャットに写真を送る
2. OpenClaw が写真を受信し、OCR でテキストを抽出する
3. AI が内容を分析:
   - 種類: 幼稚園のお知らせ
   - 内容: 3月の行事予定表
   - 振り分け先: Notion DB「幼稚園」レコード
4. Telegram に以下のようなプレビューメッセージが届く:

   📄 書類を認識しました

   種類: お知らせ（幼稚園）
   内容: 3月の行事予定表
   追加先: 幼稚園 > 行事予定
   追加内容:
     - 3/15 卒園式リハーサル
     - 3/20 卒園式
     - 3/25 修了式

   [✅ OK] [✏️ 修正] [❌ キャンセル]

5. ユーザーが [✅ OK] をタップ
6. Notion に書き込まれ、確定メッセージが届く
```

**期待される結果**: 撮影から Notion 登録完了まで30秒以内。手動操作は撮影＋OK のみ。

### シナリオ 8-2: 修正が必要な場合

**状況**: AIの分類が間違っていた（「幼稚園」ではなく「習い事」の書類だった）

**シナリオ**:

```
1. プレビューメッセージを見て、振り分け先が違うことに気づく
2. [✏️ 修正] をタップ
3. OpenClaw: 「どこを修正しますか？」
4. ユーザー: 「これは幼稚園じゃなくてスイミングスクールの書類」
5. OpenClaw が修正案を再表示:

   📄 修正しました

   種類: お知らせ（習い事）
   内容: 4月のスケジュール変更
   追加先: スイミングスクール > お知らせ
   追加内容:
     - 4月から土曜クラスの時間が10:00→10:30に変更

   [✅ OK] [✏️ 修正] [❌ キャンセル]

6. ユーザーが [✅ OK] をタップ → 確定
```

**期待される結果**: 自然言語で修正指示を出すだけで再分類される。Notion を直接開く必要がない。

### シナリオ 8-5: 既存画像のテキスト化（マイグレーション）

**状況**: Notion にはすでに大量のスキャン書類が画像としてアップロードされている。
画像だと検索できない、ページが重い、Notion の画像表示が不安定になることがある。
ただし画像自体は残したい（確定申告の提出用など原本として価値がある）。

**前提の判断基準**:
- **テキスト化すべき画像**: 医療明細、幼稚園のお手紙、通知書類など「文字情報が主体」のもの
- **テキスト化しない画像**: ビジュアライズ目的で意図的にアップロードしたもの（図表、写真など）
- → この判断は **画像ごとにユーザーが行う**（自動で一律変換しない）

**シナリオ**:

```
1. ユーザーが Telegram でマイグレーションを開始する指示を送る
   （例: 「狛江みずほ幼稚園の画像をテキスト化して」）

2. OpenClaw が該当ページの画像ブロックを一括収集する

3. 各画像に対して OCR を実行し、テキストを先に抽出しておく

4. Telegram にバッチ（5〜10件ずつ）でレビュー依頼を送信:

   📄 画像テキスト化レビュー（1/12）

   ページ: 狛江みずほ幼稚園
   [サムネイル画像]

   OCR結果:
   「12/17（水）お餅つき
     持ち物: お弁当 + タッパーウェア（3,4,5才）
     降園時間: 14時 ...」

   [✅ テキスト化] [⏭ スキップ]

5-a. [✅ テキスト化] を押した場合:
   - テキストブロックを画像の直前（または直後）に追加
   - 画像をトグル「原本画像」内に移動（折りたたみ）
   - → ページが軽量化され、テキストで検索可能になる

   変換後のNotionページ構造:
   ┌─────────────────────────────┐
   │ お餅つきについて（2025.12.2）│
   │                             │
   │ 12/17（水）お餅つき         │
   │ 持ち物: お弁当 + タッパー   │
   │ 降園時間: 14時              │
   │ ...                         │
   │                             │
   │ ▶ 原本画像                  │
   │   └─ [折りたたまれた画像]   │
   └─────────────────────────────┘

5-b. [⏭ スキップ] を押した場合:
   - 何もしない。画像はそのまま残る

6. 全件完了後、最終サマリー:

   ✅ 画像テキスト化が完了しました
   - テキスト化: 8件
   - スキップ: 4件
   - 対象ページ: 狛江みずほ幼稚園
```

**期待される結果**:
- 既存の画像は削除されず、トグル内に保持される
- テキスト版が追加され、Notion の検索で書類がヒットするようになる
- トグル折りたたみによりページの読み込みが軽くなる
- OCR精度が低い画像はスキップして手動対応できる

**追加オプション（将来検討）**:
- OCR結果を Telegram 上で編集してから反映（誤字修正）
- 「全部テキスト化」ボタンで一括承認（明らかに全件OKな場合）
- スキップした画像に後からリトライ
- 「提出用に画像を目立つ位置に残す」フラグ

| サービス | 用途 |
|---|---|
| Telegram Bot API | ユーザーとの対話（写真受信・プレビュー表示・ボタンUI） |
